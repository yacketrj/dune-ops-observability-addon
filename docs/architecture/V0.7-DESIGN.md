# v0.7.0 — SOC/OPS Operations Center Design

**Release**: v0.7.0 (was v1.0.0)
**Status**: Design
**Date**: 2026-07-07
**Depends on**: Core R3 (Prometheus Bridge API) + Core R4 (SOC Hardening)

---

## Abstract

Consolidates all existing panels into a unified operator command surface. Adds live Prometheus metrics display via the R3 `metrics.query` bridge action. Requires Core R3+R4 — cannot ship before both are deployed.

---

## ops.soc.summary

| Field | Type | Source | Description |
|---|---|---|---|
| `platformHealth` | string | Prometheus `up` query | Healthy / Degraded / Unknown |
| `bridgeRequests` | int | Console API `/metrics` | Total bridge action calls |
| `bridgeErrors` | int | Console API `/metrics` | Failed bridge action calls |
| `bridgeSuccessRate` | float | Calculated | (requests - errors) / requests * 100 |
| `metricsQueryEnabled` | bool | Core R3 availability | Whether Prometheus bridge is active |

## ops.health.prometheus (requires Core R3)

| Field | Type | Source | Description |
|---|---|---|---|
| `healthy` | bool | Prometheus health check | Up/down |
| `targets` | object | `{ active, inactive, pending, total }` | Scrape target status |
| `services` | object | Per-service status map | Each exporter/container up/down |
| `summary` | object | `{ avgCpuPercent, avgMemoryMb, totalRestarts }` | Aggregated resource usage |

## Core Dependencies

| Dependency | Release | What it provides |
|---|---|---|
| `metrics.query` bridge | Core R3 | Live Prometheus queries from addon |
| `/api/server/metrics` | Core R3 | Host-level CPU/mem/disk endpoint |
| Persistent rate limits | Core R4 | Redis-backed, survives restart |
| Per-addon CSP | Core R4 | Content-Security-Policy per addon |

## Security

- `metrics.query` validates PromQL — no unbounded cardinality, no raw metric names
- Response limited to `ADMIN_MAX_JSON_BYTES`
- Platform health computed server-side from aggregate data
- No raw Prometheus targets or scrape config exposed
